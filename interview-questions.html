What is the Event Loop? Explain Call Stack, Microtask Queue, and Macrotask Queue with an example.
Theoretical Advanced
▾
✦ Answer

JavaScript is single-threaded. The Event Loop is the mechanism that allows non-blocking async operations despite that.

  • Call Stack — Executes synchronous code LIFO (last in, first out).
  • Microtask Queue — Holds Promise.then, queueMicrotask, MutationObserver. Runs completely empty before any macrotask.
  • Macrotask Queue — Holds setTimeout, setInterval, I/O, UI paint. One macrotask runs per loop tick.
// What is the output?
console.log('1');

setTimeout(() => console.log('2'), 0);

new Promise(resolve => {
  console.log('3');
  resolve();
}).then(() => console.log('4'));

console.log('5');
1 → 3 → 5 → 4 → 2
Rule: Sync → Microtasks (all) → One Macrotask → Microtasks (all) → repeat.

Explain Closure. Write a counter factory using closure. Why does the classic loop+setTimeout bug happen and how to fix it?
Practical Tricky
▾
✦ Answer

A closure is a function that retains access to its outer lexical scope even after that scope has returned.

// Counter factory
function makeCounter(start = 0) {
  let count = start;
  return {
    increment: () => ++count,
    decrement: () => --count,
    value:     () => count,
  };
}
const c = makeCounter(10);
c.increment(); // 11
c.value();     // 11
// Classic bug — prints 5,5,5,5,5
for (var i = 0; i < 5; i++) {
  setTimeout(() => console.log(i), 100);
}

// Fix 1: use let (block scope)
for (let i = 0; i < 5; i++) {
  setTimeout(() => console.log(i), 100);
}

// Fix 2: IIFE closure
for (var i = 0; i < 5; i++) {
  ((j) => setTimeout(() => console.log(j), 100))(i);
}
Why? var is function-scoped — all callbacks share the same i. let creates a new binding per iteration.

What is Prototypal Inheritance? How does Object.create differ from class extends?
TheoreticalAdvanced
▾
✦ Answer

Every JS object has an internal [[Prototype]] link. Property lookup walks this chain until null.

// Object.create — pure prototype chain
const animal = {
  speak() { return `${this.name} makes a sound`; }
};
const dog = Object.create(animal);
dog.name = 'Rex';
dog.speak(); // "Rex makes a sound"

// class syntax (syntactic sugar over prototype)
class Animal {
  constructor(name) { this.name = name; }
  speak() { return `${this.name} speaks`; }
}
class Dog extends Animal {
  speak() { return `${this.name} barks`; }
}
Dog.prototype instanceof Animal; // false — but Dog.__proto__ === Animal
Key: class also sets up the prototype chain the same way — it's purely syntactic sugar. Unlike classical OOP, properties aren't copied; they're looked up via the chain.

Explain this binding rules. What does this refer to in arrow functions vs regular functions?
TrickyPractical
▾
✦ Answer

this is determined at call time (except arrow functions which use lexical scope).

  • Default binding: global / undefined in strict mode
  • Implicit binding: left of the dot — obj.fn() → this = obj
  • Explicit binding: call, apply, bind
  • new binding: newly created object
  • Arrow: inherits this from enclosing lexical scope, cannot be changed
const obj = {
  name: 'JS',
  regular() { return this.name; },        // 'JS'
  arrow:  () => this?.name ?? 'undefined', // undefined — lexical this
};

const fn = obj.regular;
fn();              // undefined (default binding)
fn.call(obj);      // 'JS' (explicit)
const bound = fn.bind(obj);
bound();           // 'JS'

Implement Promise.all, Promise.allSettled, and Promise.race from scratch.
PracticalAdvanced
▾
✦ Answer
// Promise.all — rejects if ANY promise rejects
Promise.myAll = (promises) => new Promise((resolve, reject) => {
  let results = [], done = 0;
  promises.forEach((p, i) =>
    Promise.resolve(p).then(val => {
      results[i] = val;
      if (++done === promises.length) resolve(results);
    }, reject)
  );
  if (!promises.length) resolve([]);
});

// Promise.allSettled — waits for all, never rejects
Promise.myAllSettled = (promises) => new Promise(resolve => {
  let results = [], done = 0;
  promises.forEach((p, i) =>
    Promise.resolve(p).then(
      value  => { results[i] = { status: 'fulfilled', value };  settle(); },
      reason => { results[i] = { status: 'rejected',  reason }; settle(); }
    )
  );
  function settle() { if (++done === promises.length) resolve(results); }
  if (!promises.length) resolve([]);
});

// Promise.race — resolves/rejects with first settled
Promise.myRace = (promises) => new Promise((resolve, reject) =>
  promises.forEach(p => Promise.resolve(p).then(resolve, reject))
);

What is Debounce vs Throttle? Implement both from scratch with use-cases.
PracticalTheoretical
▾
✦ Answer
  • Debounce — delays execution until after N ms of inactivity. Use: search input, form validation.
  • Throttle — allows execution at most once every N ms. Use: scroll, resize, button spam.
// Debounce
function debounce(fn, delay) {
  let timer;
  return function(...args) {
    clearTimeout(timer);
    timer = setTimeout(() => fn.apply(this, args), delay);
  };
}

// Throttle (leading edge)
function throttle(fn, limit) {
  let inThrottle;
  return function(...args) {
    if (!inThrottle) {
      fn.apply(this, args);
      inThrottle = true;
      setTimeout(() => (inThrottle = false), limit);
    }
  };
}

// Usage
const onSearch = debounce(val => fetchResults(val), 300);
const onScroll = throttle(() => updateHeader(), 100);

What is Currying? Implement a generic curry function that supports partial application.
PracticalAdvanced
▾
✦ Answer

Currying transforms f(a,b,c) into f(a)(b)(c). Enables partial application and function composition.

function curry(fn) {
  return function curried(...args) {
    if (args.length >= fn.length) {
      return fn.apply(this, args);
    }
    return function(...moreArgs) {
      return curried.apply(this, args.concat(moreArgs));
    };
  };
}

const add = curry((a, b, c) => a + b + c);
add(1)(2)(3);  // 6
add(1, 2)(3);  // 6
add(1)(2, 3);  // 6

// Real-world: reusable validators
const hasMinLength = curry((min, str) => str.length >= min);
const minLength8 = hasMinLength(8);
minLength8('password123'); // true

What are Generators? Implement an infinite ID generator and explain lazy evaluation.
PracticalAdvanced
▾
✦ Answer

Generators (function*) can pause and resume execution at yield points, enabling lazy sequences.

// Infinite ID generator
function* idGenerator(start = 1) {
  while (true) yield start++;
}
const gen = idGenerator();
gen.next().value; // 1
gen.next().value; // 2

// Lazy range
function* range(start, end, step = 1) {
  for (let i = start; i < end; i += step) yield i;
}
[...range(0, 10, 2)]; // [0,2,4,6,8]

// Async generator (real-world pagination)
async function* paginate(url) {
  let page = 1;
  while (true) {
    const res = await fetch(`${url}?page=${page}`).then(r => r.json());
    if (!res.data.length) break;
    yield res.data;
    page++;
  }
}

What is the difference between == and ===? Explain Type Coercion with tricky examples.
TrickyTheoretical
▾
✦ Answer

=== (strict equality) checks value AND type. == performs abstract equality with implicit coercion.

0  == ''          // true  (0 == 0)
0  == '0'         // true
'' == '0'         // false
null == undefined // true  (special case)
null == 0         // false
false == 'false'  // false
false == '0'      // true
[] == ![]          // true  (mind-bender!)
// [] == false → '' == false → 0 == 0 → true

// Type coercion in arithmetic
'5' - 3   // 2   (string coerced to number)
'5' + 3   // '53' (number coerced to string!)
+'5'      // 5   (unary + coercion)
!!''      // false (falsy)
Rule of thumb: Always use ===. Use == only when explicitly comparing null/undefined.

What are WeakMap and WeakSet? When would you use them over Map/Set?
TheoreticalAdvanced
▾
✦ Answer

WeakMap/WeakSet hold weak references to their object keys. If the object is garbage collected, entries are auto-removed.

  • Keys must be objects (not primitives)
  • Not iterable, no size property
  • Prevents memory leaks in long-lived apps
// Cache without memory leak
const cache = new WeakMap();

function processUser(user) {
  if (cache.has(user)) return cache.get(user);
  const result = heavyComputation(user);
  cache.set(user, result);
  return result;
}
// When 'user' object is GC'd, cache entry is too

// Private data pattern
const _private = new WeakMap();
class BankAccount {
  constructor(balance) { _private.set(this, { balance }); }
  getBalance() { return _private.get(this).balance; }
}

Implement a deep clone function. What are the pitfalls of JSON.parse(JSON.stringify(obj))?
PracticalTricky
▾
✦ Answer

JSON pitfalls: loses undefined, functions, Symbol, Date (becomes string), RegExp, circular refs throw.

function deepClone(obj, seen = new WeakMap()) {
  if (obj === null || typeof obj !== 'object') return obj;
  if (seen.has(obj)) return seen.get(obj);  // circular ref

  if (obj instanceof Date)   return new Date(obj);
  if (obj instanceof RegExp) return new RegExp(obj);
  if (obj instanceof Map) {
    const m = new Map();
    seen.set(obj, m);
    obj.forEach((v, k) => m.set(deepClone(k, seen), deepClone(v, seen)));
    return m;
  }
  const clone = Array.isArray(obj) ? [] : Object.create(Object.getPrototypeOf(obj));
  seen.set(obj, clone);
  for (const key of Reflect.ownKeys(obj)) {
    clone[key] = deepClone(obj[key], seen);
  }
  return clone;
}
Modern alternative: structuredClone(obj) — built-in, handles most types except functions.

Explain Hoisting. What is the Temporal Dead Zone (TDZ)?
TheoreticalTricky
▾
✦ Answer

Hoisting: Declarations are moved to the top of their scope at compile time. Initialisations are NOT.

console.log(x); // undefined (not error) — var hoisted
var x = 5;

console.log(y); // ReferenceError: TDZ
let y = 5;

// Function declarations are fully hoisted
greet(); // "Hello" — works!
function greet() { console.log('Hello'); }

// Function expressions are NOT
sayHi(); // TypeError: sayHi is not a function
var sayHi = () => console.log('Hi');
TDZ: let/const are hoisted but not initialized. Accessing them before their line = ReferenceError.

What is Memoization? Implement a generic memoize function with cache invalidation.
PracticalAdvanced
▾
✦ Answer
function memoize(fn, { ttl } = {}) {
  const cache = new Map();
  return function(...args) {
    const key = JSON.stringify(args);
    if (cache.has(key)) {
      const { value, expiry } = cache.get(key);
      if (!expiry || Date.now() < expiry) return value;
    }
    const value = fn.apply(this, args);
    cache.set(key, { value, expiry: ttl ? Date.now() + ttl : null });
    return value;
  };
}

// Usage
const expFib = memoize((n) =>
  n <= 1 ? n : expFib(n - 1) + expFib(n - 2)
);
expFib(40); // instant after first call

const cachedFetch = memoize(url => fetch(url).then(r => r.json()), { ttl: 60000 });

What are JavaScript Memory Leaks? Name common causes and how to detect them.
TheoreticalAdvanced
▾
✦ Answer

A memory leak is unreleased memory that the app no longer needs but can't be GC'd.

  • Global variables — accidental globals via missing var/let/const
  • Forgotten timers/intervals — setInterval not cleared
  • Detached DOM nodes — removed from DOM but referenced in JS
  • Closures — holding large objects in scope unnecessarily
  • Event listeners — not removed when component unmounts
// Leak: setInterval keeps 'data' alive
function startPolling() {
  const data = new Array(100000).fill('data');
  setInterval(() => console.log(data.length), 1000);
  // Never clears — data never GC'd
}

// Fix: store id and clear
let timerId;
function start() { timerId = setInterval(poll, 1000); }
function stop()  { clearInterval(timerId); }
Detection: Chrome DevTools → Memory tab → Heap Snapshot / Allocation instrumentation on timeline.

function reactive(target) {
  return new Proxy(target, {
    get(obj, prop) {
      console.log(`GET: ${prop} = ${obj[prop]}`);
      return typeof obj[prop] === 'object'
        ? reactive(obj[prop])   // deep reactivity
        : obj[prop];
    },
    set(obj, prop, value) {
      console.log(`SET: ${prop} = ${value}`);
      obj[prop] = value;
      return true; // required by spec
    },
    deleteProperty(obj, prop) {
      console.log(`DELETE: ${prop}`);
      delete obj[prop];
      return true;
    }
  });
}
const state = reactive({ count: 0 });
state.count++;    // GET: count = 0 | SET: count = 1

Implement a pub/sub (EventEmitter) system with on, off, emit, and once.
PracticalAdvanced
▾
✦ Answer
class EventEmitter {
  #events = new Map();

  on(event, listener) {
    if (!this.#events.has(event)) this.#events.set(event, new Set());
    this.#events.get(event).add(listener);
    return () => this.off(event, listener); // unsubscribe fn
  }

  off(event, listener) {
    this.#events.get(event)?.delete(listener);
  }

  emit(event, ...args) {
    this.#events.get(event)?.forEach(fn => fn(...args));
  }

  once(event, listener) {
    const wrapper = (...args) => { listener(...args); this.off(event, wrapper); };
    return this.on(event, wrapper);
  }
}

const emitter = new EventEmitter();
const unsub = emitter.on('data', v => console.log(v));
emitter.emit('data', 42); // 42
unsub();                 // unsubscribed

const ID = Symbol('id');

}

What is the difference between Object.freeze, Object.seal, and const?
TrickyTheoretical
▾
✦ Answer
  • const — the binding can't be reassigned. The object itself is still mutable.
  • Object.seal — can't add/delete properties, but can modify existing values.
  • Object.freeze — can't add/delete/modify properties. Shallow only.
const obj = { a: 1 };
obj.a = 2;        // OK — const doesn't freeze
obj = {};          // TypeError — reassignment blocked

const sealed = Object.seal({ x: 1 });
sealed.x = 99;   // OK
sealed.y = 2;    // silently fails (TypeError in strict)

const frozen = Object.freeze({ x: 1, nested: { y: 2 } });
frozen.x = 99;          // silently fails
frozen.nested.y = 99;   // WORKS — freeze is shallow!

// Deep freeze
function deepFreeze(obj) {
  Object.getOwnPropertyNames(obj).forEach(name => {
    if (typeof obj[name] === 'object' && obj[name]) deepFreeze(obj[name]);
  });
  return Object.freeze(obj);
}

What is typeof null === 'object'? Explain this bug and how to safely check for null.
TrickyTheoretical
▾
✦ Answer

This is a historical bug from JavaScript's first version. In the original implementation, values were stored as type tags. null had the tag 000 (same as objects), so typeof null returns 'object'. It was never fixed to maintain backwards compatibility.

typeof {}          // 'object'
typeof function(){} // 'function'

Implement a rate limiter that allows at most N requests per second using the Token Bucket algorithm.
PracticalAdvanced
▾
✦ Answer
class RateLimiter {
  #tokens;
  #maxTokens;
  #refillRate;   // tokens per ms
  #lastRefill;

  constructor(maxPerSecond) {
    this.#tokens    = maxPerSecond;
    this.#maxTokens = maxPerSecond;
    this.#refillRate = maxPerSecond / 1000;
    this.#lastRefill = Date.now();
  }

  #refill() {
    const now = Date.now();
    const elapsed = now - this.#lastRefill;
    this.#tokens = Math.min(
      this.#maxTokens,
      this.#tokens + elapsed * this.#refillRate
    );
    this.#lastRefill = now;
  }

  tryConsume() {
    this.#refill();
    if (this.#tokens < 1) return false;
    this.#tokens--;
    return true;
  }
}

const limiter = new RateLimiter(5); // 5 req/sec
if (limiter.tryConsume()) { callAPI(); }
else { showError('Rate limit exceeded'); }

▾
✦ Answer
  • localStorage — persists after tab/browser close, ~5MB, same origin only
  • sessionStorage — cleared on tab close, ~5MB, per-tab scope
  • Cookies — sent with every HTTP request, can be HttpOnly, Secure, SameSite
// localStorage / sessionStorage
localStorage.setItem('theme', 'dark');
localStorage.getItem('theme');
localStorage.removeItem('theme');

// Secure cookie (set server-side)
// Set-Cookie: token=xyz; HttpOnly; Secure; SameSite=Strict; Path=/

// Security matrix:
// | Feature        | localStorage | Cookie       |
// | XSS risk       | HIGH ✗       | HttpOnly ✓   |
// | CSRF risk      | N/A          | SameSite ✓   |
// | Sent to server | No           | Yes (auto)   |
// | Expiry control | Manual JS    | Max-Age/Expires |
Best practice: Store JWT in HttpOnly cookie, not localStorage. XSS can't read HttpOnly cookies.

Explain Content Security Policy (CSP). How do you prevent XSS using HTTP headers and HTML?
PracticalAdvanced
▾
✦ Answer

CSP is a browser security feature that restricts which resources can be loaded, preventing XSS injection.

<!-- Via meta tag (limited) -->
<meta
  http-equiv="Content-Security-Policy"
  content="default-src 'self'; script-src 'self' 'nonce-{random}'; style-src 'self'"
/>

<!-- Nonce-based scripts (blocks injected scripts) -->
<script nonce="random123">
  // Only this runs — even if attacker injects script tags
</script>

<!-- Via HTTP header (preferred) -->
Content-Security-Policy:
  default-src 'self';
  script-src 'self' https://trusted.cdn.com 'nonce-abc123';
  img-src 'self' data: https:;
  frame-ancestors 'none';
  upgrade-insecure-requests;
Also set: X-Frame-Options: DENY, X-Content-Type-Options: nosniff, Referrer-Policy: strict-origin-when-cross-origin.
